# Agent Instructions

- Отвечай пользователю на русском языке, если он сам не попросил иначе.
- Перед изменениями изучай существующую структуру проекта и используй локальные паттерны.
- Не добавляй новые библиотеки без явной пользы для задачи.
- Не трогай несвязанные файлы и не откатывай чужие изменения.
- Для долгих задач веди состояние в `TASKS.md` или `PLAN.md`.
- Важные архитектурные решения записывай в `DECISIONS.md`.
- Эксперименты, неудачные подходы и результаты проверок записывай в `EXPERIMENTS.md`.
- Устойчивые предпочтения пользователя и проекта записывай в `AGENT_MEMORY.md`.
- После изменений в 3D/frontend-поверхности запускай `npm run qa`, если окружение позволяет.
- Для проверки доступности инструментов используй `npm run check:tools`.
- Если задача касается визуального результата, проверяй, что canvas/сцена не пустые и нет ошибок консоли.
- Не превращай рабочий инструмент в лендинг, если пользователь просит приложение или сцену.
- Для сложных задач думай как Orchestrator: разделяй работу на Architect, Builder, QA, Memory Curator и Safety Reviewer.
- Перед длинной, рискованной или многофайловой задачей создавай checkpoint через `npm run agent:checkpoint -- create "описание"`.
- Для оценки состояния агента используй `npm run agent:evals`.
- Для памяти используй активное управление: полезные записи отмечай `memory:db -- touch`, слабые записи просматривай через `memory:db -- prune-candidates`.
- Перед долгими, дорогими, сетевыми или потенциально опасными действиями спрашивай подтверждение пользователя.
- Когда показываешь или описываешь текущую 3D-сцену, используй только финальные артефакты `output/preview.png`, `output/scene.blend`, `output/scene.glb`.
- Промежуточные или неудачные рендеры всегда называй draft/attempt и сохраняй отдельно через `npm run scene:attempt -- snapshot "описание"`.
- Перед финальным ответом по 3D-сцене запускай `npm run scene:status` или заново открывай `output/preview.png`.
